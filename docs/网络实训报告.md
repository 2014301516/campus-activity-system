# 校园活动管理系统 — 网络综合开发实训报告

## 一、项目简介

**项目名称**：校园活动管理系统（Campus Activity Management System）

**项目内容**：本系统是一个面向高校师生的校园活动管理平台，支持学生浏览活动、在线报名、活动签到签退、活动评价，活动组织者发布和管理活动，以及管理员进行用户管理、活动审核、分类管理和公告发布等操作。系统采用前后端分离架构，前端使用 Vue 3 + Element Plus 构建现代化用户界面，后端基于 Spring Boot + MyBatis-Plus 提供 RESTful API 服务。

**技术栈**：
- 前端：Vue 3 + Element Plus + Vue Router + Pinia + Axios + ECharts
- 后端：Spring Boot 2.7 + MyBatis-Plus + Spring Security + JWT
- 数据库：MySQL 8.0
- 开发工具：VS Code / IntelliJ IDEA / Git / Postman

---

## 二、项目任务

本项目需要完成以下核心任务：
1. 设计并实现校园活动管理系统，包括用户管理、活动管理、报名管理、签到管理和评价管理等功能模块
2. 系统需包含 7 个页面，7 张核心数据表，3 种用户角色
3. 系统支持完整的增删改查操作，具备分页查询、搜索筛选、表单校验等功能
4. 后端实现分层架构（Controller-Service-Mapper），使用 JWT 进行身份认证
5. 前端使用 Vue 3 Composition API 开发，采用 Element Plus UI 组件库

## 三、项目实现

### 3.1 需求分析

本系统旨在解决校园内活动管理的信息化需求，主要功能需求包括：

**功能需求**：
- 用户注册与登录：支持学号注册，JWT Token 认证
- 活动浏览：活动列表展示，支持分类筛选、关键词搜索、排序
- 活动报名：学生可报名活动，系统自动校验人数上限
- 签到签退：报名用户可在活动时间段内签到和签退
- 活动评价：参与活动的用户可对活动进行星级评分和文字评价
- 活动管理：组织者可发布、编辑、删除活动
- 活动审核：管理员审核待发布的活动
- 用户管理：管理员管理用户账号、修改角色和状态
- 分类管理：管理员管理活动分类
- 公告管理：管理员发布和管理系统公告
- 数据统计：仪表盘展示活动数量、用户数量、分类统计等

**角色需求**：
- 学生（student）：浏览活动、报名、签到、评价
- 组织者（organizer）：发布活动、管理报名、查看签到
- 管理员（admin）：用户管理、活动审核、分类管理、公告管理、数据统计

### 3.5 难点分析

#### 难点1：JWT 认证与 Spring Security 集成

**问题描述**：需要将 JWT Token 认证机制与 Spring Security 框架集成，实现无状态的用户认证。

**解决方案**：
- 创建 `JwtAuthFilter` 继承 `OncePerRequestFilter`，从请求头提取 Bearer Token
- 解析 Token 后将用户信息（userId、role）设置到 `SecurityContextHolder`
- 配置 `SecurityConfig` 使用无状态会话策略（`SessionCreationPolicy.STATELESS`）
- 将 JWT 过滤器添加到 `UsernamePasswordAuthenticationFilter` 之前

#### 难点2：前后端分离跨域问题

**问题描述**：前端（localhost:5173）与后端（localhost:8080）不同端口，存在跨域问题。

**解决方案**：
- 后端配置 `CorsConfig`，使用 Spring 的 `CorsFilter` 允许所有来源
- 前端开发环境配置 Vite proxy，将 `/api` 请求代理到 `localhost:8080`
- 双重保障确保跨域访问正常

#### 难点3：活动状态流转管理

**问题描述**：活动从创建到结束有多个状态（草稿→待审核→已通过→进行中→已结束），需要合理管理状态转换。

**解决方案**：
- 在 `ActivityServiceImpl` 中对每个操作添加状态校验
- 例如：只有"待审核"状态才能被审核，只有"已通过"或"进行中"状态才能报名
- 前端在操作按钮上做好状态判断和禁用逻辑

#### 难点4：报名并发控制

**问题描述**：多个学生同时报名同一活动时，可能出现超员问题。

**解决方案**：
- 在 `register` 方法上使用 `@Transactional` 注解保证原子性
- 报名前检查 `currentParticipants < maxParticipants`
- 成功后 `currentParticipants + 1`

# 校园活动管理系统 前后端通信说明

> 说明：本文档只讨论 `e:\keshe\campus-activity-system`，不涉及其他项目。
>
> 用途：帮助我快速理解这个项目的前后端通信流程，方便答辩时讲解。

---

## 1. 一句话理解

这个项目的前后端通信方式可以概括为：

**前端页面发请求 -> Axios 统一封装 -> Vite 代理到后端 -> Spring Boot Controller 接收 -> Service 处理业务 -> Mapper 操作数据库 -> Result 统一返回给前端**

所以它本质上是一个典型的：

- 前后端分离项目
- 基于 RESTful API 通信
- 使用 JWT 做身份认证

## 2. 前端是如何发请求的

前端请求的核心文件在：

- [request.js](file:///e:/keshe/campus-activity-system/frontend/src/api/request.js)
- [index.js](file:///e:/keshe/campus-activity-system/frontend/src/api/index.js)

### 2.1 `request.js` 的作用

[request.js](file:///e:/keshe/campus-activity-system/frontend/src/api/request.js) 是对 Axios 的统一封装。

它主要负责 4 件事：

1. 设置请求基础地址 `baseURL: '/api'`
2. 设置请求超时时间
3. 在请求头里自动加上 token
4. 对返回结果做统一错误处理

它的关键代码逻辑可以这样理解：

```js
const request = axios.create({
  baseURL: '/api',
  timeout: 15000
})
```

含义是：

- 以后所有接口默认都从 `/api` 开始
- 页面里不用每次都写完整地址

### 2.2 请求拦截器做了什么

在请求发出去之前，请求拦截器会先执行：

```js
const token = localStorage.getItem('token')
if (token) {
  config.headers['Authorization'] = `Bearer ${token}`
}
```

这段逻辑的作用是：

- 从本地存储中取出登录后的 token
- 自动放进请求头
- 后端就能识别当前请求是谁发来的

所以前端页面本身不需要每次手动写：

```js
headers: { Authorization: 'Bearer xxx' }
```

因为 `request.js` 已经统一处理了。

请求拦截器在源码中的位置是：

- [request.js:L9-L19](file:///e:/keshe/campus-activity-system/frontend/src/api/request.js#L9-L19)

它对应的写法是：

```js
request.interceptors.request.use(...)
```

这里的 `request` 是通过 `axios.create(...)` 创建出来的 Axios 实例。

可以这样理解：

- 页面想发请求时，不是直接用原始 Axios
- 而是统一使用这个 `request` 实例
- 只要使用这个实例，请求发出去之前都会先经过请求拦截器

所以请求拦截器的执行时机是：

```text
页面调用接口
-> request 实例准备发请求
-> 先进入请求拦截器
-> 自动加 token
-> 再真正发给后端
```

### 2.3 响应拦截器做了什么

后端返回结果后，响应拦截器会统一判断：

```js
if (res.code !== 200) {
  ElMessage.error(res.message || '请求失败')
  if (res.code === 401) {
    localStorage.removeItem('token')
    localStorage.removeItem('userInfo')
    window.location.hash = '#/login'
  }
  return Promise.reject(new Error(res.message))
}
return res
```

作用是：

- 如果后端返回失败，前端统一弹出错误提示
- 如果是 `401`，说明登录失效，就自动清理登录信息并跳回登录页
- 如果成功，就把结果继续返回给页面使用

响应拦截器在源码中的位置是：

- [request.js:L21-L40](file:///e:/keshe/campus-activity-system/frontend/src/api/request.js#L21-L40)

它对应的写法是：

```js
request.interceptors.response.use(...)
```

可以这样理解：

- 请求先发给后端
- 后端返回结果后
- 结果不会立刻直接给页面
- 而是先经过响应拦截器统一处理

所以响应拦截器的执行时机是：

```text
后端返回结果
-> 先进入响应拦截器
-> 统一判断 code 是否成功
-> 成功则返回给页面
-> 失败则统一提示错误
```

### 2.4 为什么拦截器要写在 `request.js`

因为 `request.js` 是整个前端项目的统一请求入口。

项目里的接口最终都会走：

- [api/index.js](file:///e:/keshe/campus-activity-system/frontend/src/api/index.js)

而 `api/index.js` 里面所有模块都依赖：

- [request.js](file:///e:/keshe/campus-activity-system/frontend/src/api/request.js)

也就是说：

- `userApi` 用的是 `request`
- `activityApi` 用的是 `request`
- `registrationApi` 用的是 `request`
- `signInApi` 用的是 `request`

只要它们都通过同一个 `request` 实例发请求，就会自动共享：

- token 添加逻辑
- 错误提示逻辑
- 401 失效处理逻辑

如果不这么做，而是每个页面自己写 Axios，那么就会出现：

- 每个页面都要自己加 token
- 每个页面都要自己处理错误
- 代码重复很多
- 后期维护很麻烦

所以把拦截器放在 `request.js`，本质上就是为了：

1. 统一请求行为
2. 减少重复代码
3. 让页面层只关注业务逻辑

### 2.5 页面、接口封装、拦截器三者的关系

这三层关系可以这样记：

```text
Vue 页面
-> 调用 api/index.js 里的接口方法
-> 接口方法内部调用 request.js
-> request.js 的请求拦截器先执行
-> 请求发到后端
-> 后端返回结果
-> request.js 的响应拦截器处理结果
-> 最终结果再回到页面
```

例如：

```text
MyActivities.vue
-> registrationApi.getMyRegistrations()
-> request.get('/my-registrations')
-> 请求拦截器自动加 token
-> 后端返回报名数据
-> 响应拦截器判断 code
-> 页面拿到结果并渲染表格
```

## 3. 前端接口是如何组织的

接口集中定义在：

- [api/index.js](file:///e:/keshe/campus-activity-system/frontend/src/api/index.js)

这里把接口按业务划分成多个模块：

- `userApi`
- `activityApi`
- `registrationApi`
- `signInApi`
- `reviewApi`
- `categoryApi`
- `noticeApi`
- `dashboardApi`
- `adminApi`

这样做的好处是：

- 页面代码更清楚
- 不需要每个页面自己拼接口地址
- 后期维护更方便

例如：

```js
export const activityApi = {
  getList: (params) => request.get('/activity/list', { params }),
  getDetail: (id) => request.get(`/activity/${id}`),
  create: (data) => request.post('/activity', data),
  update: (id, data) => request.put(`/activity/${id}`, data),
  delete: (id) => request.delete(`/activity/${id}`)
}
```

含义是：

- 活动列表、详情、创建、更新、删除都统一封装好了
- 页面只管调用方法，不管底层 URL 和请求方式

## 4. 为什么本地前后端能方便联调

原因在于：

- 前端开发服务运行在 `5173`
- 后端运行在 `8080`
- [vite.config.js](file:///e:/keshe/campus-activity-system/frontend/vite.config.js) 中配置了代理

关键配置如下：

```js
server: {
  port: 5173,
  proxy: {
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true
    }
  }
}
```

这表示：

- 当前端请求 `/api/...`
- Vite 会自动把它转发到 `http://localhost:8080`

例如前端代码写的是：

```js
request.get('/activity/list')
```

真实开发环境下会被转发成：

```text
http://localhost:8080/api/activity/list
```

### 4.1 为什么这样方便

这样做主要有三个好处：

1. 不用在前端代码里到处写完整后端地址
2. 开发时更容易处理跨域问题
3. 前后端本地联调更顺畅

### 4.2 什么是跨域

当前端页面运行在：

- `http://localhost:5173`

后端接口运行在：

- `http://localhost:8080`

因为端口不同，浏览器会认为它们不是同一个源，所以直接请求时可能出现跨域问题。

配置代理后：

- 浏览器看到的还是前端自己的 `/api`
- 实际转发由 Vite 完成

所以开发体验会更好。

## 6. 后端收到请求后会做什么

后端处理流程大致如下：

```text
Controller 接收请求
-> Service 处理业务逻辑
-> Mapper 查询或更新数据库
-> Result 封装结果
-> 返回给前端
```

### 6.1 Controller 的职责

Controller 负责：

- 接收前端参数
- 调用对应的 Service
- 把结果返回给前端

例如在 [ActivityController.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/controller/ActivityController.java#L31-L75) 中：

- `list()` 负责接收活动查询条件
- `detail()` 负责接收活动 ID
- `create()` 负责接收活动表单数据

### 6.2 Service 的职责

Service 才是真正写业务规则的地方。

例如：

- `ActivityServiceImpl` 处理活动分页、创建、修改、删除、审核和状态流转
- `RegistrationServiceImpl` 处理报名和取消报名
- `SignInServiceImpl` 处理签到、签退和签到状态查询

### 6.3 Mapper 的职责

Mapper 主要负责和数据库打交道。

比如：

- `ActivityMapper`
- `RegistrationMapper`
- `SignInMapper`

它们会执行增删改查，把数据库中的数据查出来或更新进去。

## 7. 登录后为什么后端知道当前用户是谁

这是通过 `JWT + Spring Security` 实现的。

### 7.1 登录时 token 从哪里来

登录接口在：

- [UserController.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/controller/UserController.java#L41-L45)

真正生成 token 的逻辑在：

- [UserServiceImpl.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/service/impl/UserServiceImpl.java)
- [JwtUtil.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/util/JwtUtil.java)

登录时后端会：

1. 查用户名
2. 校验密码
3. 检查账号状态
4. 生成 token

token 中存了：

- 用户 ID
- 用户名
- 角色

### 7.2 前端如何使用 token

前端登录成功后，会把 token 保存到本地存储中。

之后每次发请求，请求拦截器都会自动把它带到请求头中：

```text
Authorization: Bearer xxxxx
```

### 7.3 后端如何解析 token

后端使用：

- [JwtAuthFilter.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/config/JwtAuthFilter.java)

这个过滤器会：

1. 从请求头中取出 `Authorization`
2. 检查是不是 `Bearer token`
3. 校验 token 是否有效
4. 解析出用户 ID 和角色
5. 放进 Spring Security 上下文

这样后端后续代码就能知道当前用户是谁。

## 8. 后端如何做权限控制

权限配置主要在：

- [SecurityConfig.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/config/SecurityConfig.java)

这里定义了哪些接口能公开访问，哪些接口必须登录，哪些接口只允许特定角色访问。

例如：

- 注册和登录接口公开
- 活动列表和活动详情可以公开访问
- 管理员接口 `/api/admin/**` 只有管理员能访问
- 创建、更新、删除活动只允许组织者和管理员访问
- 其余接口默认必须登录

这就实现了“不同角色访问不同接口”的控制。

---

## 9. 后端如何获取当前登录用户

工具类在：

- [SecurityUtil.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/util/SecurityUtil.java)

它可以做两件事：

1. 获取当前登录用户 ID
2. 获取当前登录用户角色

比如：

- `getCurrentUserId()`
- `getCurrentUserRole()`

这样 Service 或 Controller 就能知道：

- 这次请求是谁发起的
- 这个人是什么角色

例如创建活动时，就会先取当前用户 ID，再把这个 ID 作为组织者 ID 使用。

## 10. 前后端约定的返回格式

后端统一使用：

- [Result.java](file:///e:/keshe/campus-activity-system/backend/src/main/java/com/cas/common/Result.java)

返回格式大致如下：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

常见字段含义：

- `code`：状态码
- `message`：提示信息
- `data`：真正返回的数据

前端响应拦截器会根据 `code` 判断：

- `200`：成功
- `401`：未登录或 token 失效
- 其他：失败并提示错误

所以整个项目前后端之间不是随便返回数据，而是遵循统一格式。

## 11. 一个完整例子：学生报名活动

以“学生报名活动”为例，通信过程如下：

### 第一步：前端触发请求

学生在页面点击“报名”按钮。

页面调用：

- `registrationApi.register(activityId)`

它对应：

```js
request.post(`/activity/${activityId}/register`)
```

### 第二步：前端自动加 token

`request.js` 请求拦截器会自动把 token 加入请求头。

### 第三步：Vite 代理转发请求

前端请求的是：

```text
/api/activity/{activityId}/register
```

Vite 会把它转发到：

```text
http://localhost:8080/api/activity/{activityId}/register
```

### 第四步：后端接收请求

后端 `RegistrationController` 接收请求。

### 第五步：后端识别当前用户

`JwtAuthFilter` 解析 token，拿到当前学生 ID 和角色。

### 第六步：业务层处理报名逻辑

`RegistrationServiceImpl` 会判断：

- 用户是否已登录
- 是否是学生角色
- 是否重复报名
- 活动是否允许报名
- 活动人数是否已满

如果满足条件，就写入报名记录，并更新活动报名人数。

### 第七步：后端统一返回

后端返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": ...
}
```

### 第八步：前端更新页面

前端拿到成功结果后：

- 弹出成功提示
- 刷新活动详情或我的报名列表

## 12. 一个完整例子：学生查看我的报名

通信过程如下：

1. 前端页面加载“我的报名”
2. 调用 `registrationApi.getMyRegistrations()`
3. 请求 `GET /api/my-registrations`
4. 请求自动带上 token
5. 后端通过 `SecurityUtil.getCurrentUserId()` 获取当前学生 ID
6. 查询这个学生对应的报名记录
7. 返回报名列表给前端
8. 前端展示在 `MyActivities.vue`

所以“我的报名”并不是前端本地缓存出来的，而是真实请求后端得到的数据。

## 13. 项目前后端通信的特点

总结起来，这个项目的前后端通信有以下特点：

### 13.1 接口调用统一

- 所有请求都通过 `request.js`
- 所有业务接口都集中在 `api/index.js`

### 13.2 登录态统一

- token 保存在本地
- 请求时自动携带
- 401 时自动清理并跳登录

### 13.3 权限控制清晰

- 前端控制页面访问
- 后端控制接口访问
- JWT 负责识别身份

### 13.4 返回格式统一

- 后端统一 `Result`
- 前端统一判断 `code`

### 13.5 本地联调方便

- Vite 代理解决开发阶段前后端地址和跨域问题

## 14. 答辩时可以怎么讲

如果老师问“你们项目是怎么实现前后端通信的”，可以这样回答：

我们项目采用前后端分离架构，前端通过 Axios 请求后端 RESTful 接口。为了便于管理，我把请求统一封装在 `request.js` 中，并在请求拦截器中自动携带 JWT token。开发环境下，前端通过 Vite 代理把 `/api` 请求转发到 Spring Boot 后端。后端使用 Spring Security 和 JWT 过滤器识别当前用户身份，再由 Controller、Service、Mapper 完成业务处理和数据库操作，最后统一用 `Result` 格式把 `code`、`message` 和 `data` 返回给前端。这样就实现了登录鉴权、角色控制和前后端联调。

# 命名规范说明 — 每个目录和文件为什么这样命名

---

## 一、根目录命名

### 项目根目录

```
campus-activity-system/
```

**为什么这样命名**：英文小写 + 连字符分隔（kebab-case）。这种命名在 URL、Git 仓库名、命令行操作中都安全——不会因为空格或大小写在跨平台时出问题。

### 为什么分 backend / frontend

```
backend/     ← Spring Boot 后端
frontend/    ← Vue 3 前端
```

前后端分离架构的行业惯例。两个目录可以独立部署、独立构建、由不同的人维护。如果混在一起，前端 npm 和后端 Maven 会产生构建冲突。

## 二、后端包名 `com.cas`

```
com.cas
```

| 部分 | 含义 | 为什么 |
|------|------|--------|
| `com` | 公司/组织域名的倒序 | Java 世界约定：用域名倒序保证全球唯一。`com.cas` = Campus Activity System |
| `cas` | 项目缩写 | 简短好记，不用写全称 `campus-activity-system`（过长） |

> 行业惯例：即使不是公司项目也遵循域名倒序，防止与其他人的代码包名冲突。

## 三、后端目录结构

### 整体包结构

```
com.cas
├── common/        # 公共
├── config/        # 配置
├── controller/    # 控制器
├── service/       # 服务接口
│   └── impl/      # 服务实现
├── mapper/        # 数据访问
├── entity/        # 实体
├── dto/           # 数据传输对象
└── util/          # 工具
```

### common/ — 为什么叫 common

**common = 公共的、共用的**。这两个文件被所有 Controller 依赖，放这里表示"全项目通用"。

| 文件 | 命名逻辑 |
|------|----------|
| `Result.java` | Result = 结果。所有接口返回的都是"操作结果"，名字直指用途 |
| `GlobalExceptionHandler.java` | Global = 全局，Exception = 异常，Handler = 处理器。三个词精确描述了它干什么 |

### config/ — 为什么叫 config

**config = configuration 的缩写**。Spring 社区惯例——所有 `@Configuration` 类都放这个包。

| 文件 | 命名逻辑 |
|------|----------|
| `SecurityConfig.java` | Security（安全）+ Config（配置）。管用户认证和权限 |
| `JwtAuthFilter.java` | Jwt + Auth（认证）+ Filter（过滤器）。JWT 认证过滤器 |
| `CorsConfig.java` | CORS = Cross-Origin Resource Sharing 的缩写。跨域 |
| `MyBatisPlusConfig.java` | 框架名 + Config。MyBatis-Plus 的分页插件配置 |
| `JacksonConfig.java` | 库名 + Config。Jackson 的日期序列化配置 |

> 命名规律：`{负责什么}Config` 或 `{做什么}Filter`

### controller/ — 为什么叫 controller

**Controller = 控制器**。来自 MVC 设计模式（Model-View-Controller），负责控制请求的流向。

| 文件 | 命名逻辑 |
|------|----------|
| `UserController.java` | 管用户注册登录 |
| `ActivityController.java` | 管活动的增删改查 |
| `CategoryController.java` | 管活动分类 |
| `RegistrationController.java` | 管报名 |
| `SignInController.java` | 管签到签退 |
| `ReviewController.java` | 管评价 |
| `NoticeController.java` | 管公告 |
| `AdminController.java` | 管管理员专属操作 |
| `DashboardController.java` | 管数据统计仪表盘 |

> 命名规律：`{管什么}Controller`。一个 Controller 对应一个业务领域，不会把所有接口塞进一个文件。

### service/ + service/impl/ — 为什么用接口+实现分离

```
service/
├── UserService.java            ← 接口：定义做什么
└── impl/
    └── UserServiceImpl.java    ← 实现：定义怎么做
```

| 命名 | 逻辑 |
|------|------|
| `UserService` | Service 后缀表明这是服务层 |
| `UserServiceImpl` | Impl = Implementation 的缩写。表示这是 `UserService` 接口的实现 |

**为什么多此一举**：Spring 的依赖注入基于接口，Controller 引用的是 `UserService` 接口而不是 `UserServiceImpl` 具体类。将来换实现只需新写一个 impl，Controller 不用改。

### mapper/ — 为什么叫 mapper

**Mapper = 映射器**。MyBatis 的核心概念：把 Java 对象和数据库行互相映射。每个 Mapper 对应一张数据库表。

| 文件 | 对应表 | 命名逻辑 |
|------|--------|----------|
| `UserMapper.java` | users | User 实体 → users 表 |
| `ActivityMapper.java` | activities | Activity 实体 → activities 表 |

> 命名规律：`{实体名}Mapper`。全部继承 `BaseMapper<T>`。

### entity/ — 为什么叫 entity

**Entity = 实体**。来自数据库建模术语：一个实体对应现实世界中的一个"事物"（用户、活动、报名记录等），在数据库里就是一行数据。

| 文件 | 对应表 |
|------|--------|
| `User.java` | users |
| `Activity.java` | activities |
| `Category.java` | categories |
| `Registration.java` | registrations |
| `SignIn.java` | sign_ins |
| `Review.java` | reviews |
| `Notice.java` | notices |

> 命名规律：单数名词，跟数据库表名对应（`users` 表 → `User` 类）。Java 类名用大驼峰（PascalCase），数据库表名用小写+下划线（snake_case），MyBatis-Plus 自动转换。

### 数据库表名 vs 实体类名

| 数据库表（snake_case） | Java 实体（PascalCase） | 为什么不同 |
|------------------------|------------------------|------------|
| `users` | `User` | MySQL 惯例用小写+下划线，Java 惯例用大驼峰 |
| `sign_ins` | `SignIn` | MyBatis-Plus 自动做 `map-underscore-to-camel-case` 映射 |
| `real_name` → `realName` | 字段名也是自动转换的 |

### dto/ — 为什么叫 dto

**DTO = Data Transfer Object 的缩写**。专用于数据传输，不映射数据库。

| 文件 | 传什么 |
|------|--------|
| `LoginDTO.java` | 登录时传用户名+密码 |
| `RegisterDTO.java` | 注册时传用户信息 |
| `ActivitySaveDTO.java` | 创建/编辑活动时传活动信息 |
| `ActivityQueryDTO.java` | 查询活动列表时传筛选条件 |

> 命名规律：`{操作场景}DTO`。与 entity 严格区分：entity 管数据库映射，dto 管 HTTP 传输。

### util/ — 为什么叫 util

**util = utility 的缩写**。放工具类——不归属任何业务模块、被各处调用的通用代码。

| 文件 | 命名逻辑 |
|------|----------|
| `JwtUtil.java` | Jwt（技术名）+ Util（工具） |
| `SecurityUtil.java` | Security（领域）+ Util（工具） |

> 命名规律：`{负责什么}Util`

## 四、前端文件命名

### 目录结构

```
frontend/src/
├── views/          # 页面
├── router/         # 路由
├── api/            # 接口请求
├── store/          # 状态管理
├── components/     # 公共组件（目前空）
└── utils/          # 工具函数（目前空）
```

### views/ — 为什么叫 views

**Views = 视图**。同样来自 MVC 模式，ViewController → View，用户直接看到的部分。

| 文件 | 路由 | 为什么这样命名 |
|------|------|---------------|
| `Login.vue` | `/login` | Login = 登录。页面名和路由名一致，一眼对应 |
| `Home.vue` | `/home` | Home = 首页。约定俗成 |
| `ActivityDetail.vue` | `/activity/:id` | Activity + Detail = 活动详情。名词+名词模式 |
| `MyActivities.vue` | `/my-activities` | My + Activities = 我的活动。体现"个人"属性 |
| `ActivityManage.vue` | `/manage` | Activity + Manage = 活动管理。Manage 表示增删改查 |
| `Admin.vue` | `/admin` | Admin = 管理员页面。简洁，因为已经隐含了管理含义 |
| `Profile.vue` | `/profile` | Profile = 个人中心/个人资料。通用术语 |

> 命名规律：Vue 组件用大驼峰（PascalCase），路由用小写/连字符（kebab-case）。`ActivityDetail` 是两个单词拼成一个组件名，`/activity/:id` 是嵌套路由路径。

### router/index.js — 路由文件

| 部分 | 为什么 |
|------|--------|
| `router/` | 放路由相关代码的目录 |
| `index.js` | 目录下的主文件——import 时写 `@/router` 自动找到它。不需要写 `@/router/index.js` |

### api/ — 接口请求层

| 文件 | 为什么 |
|------|--------|
| `request.js` | 创建 axios 实例、配置拦截器——它定义的是"请求的通用规则" |
| `index.js` | 定义所有接口函数——它列出"我们有哪些请求可以发" |

**为什么叫 request 而不是 axios**：如果将来换 fetch 或别的库，文件名不用改。文件名描述**做什么**，不描述**用什么库**。

**为什么叫 index.js**：同上——`import { userApi } from '@/api'` 简洁。

### store/auth.js — 状态管理

| 部分 | 为什么 |
|------|--------|
| `store/` | Pinia 取名 "store"，对应目录也用 store |
| `auth.js` | auth = authentication 缩写。这个 Store 存的是认证相关状态（token、用户信息）。如果以后需要其他 Store（如购物车），可以新建 `cart.js` |

### main.js 和 App.vue

| 文件 | 为什么 |
|------|--------|
| `main.js` | Vue 惯例——入口文件叫 main。创建 app 实例、安装插件、挂载到 `#app` |
| `App.vue` | Vue 惯例——根组件叫 App。所有页面的外壳 |

### vite.config.js

| 部分 | 为什么 |
|------|--------|
| `vite.config.js` | 工具名 + config。Vite 启动时自动读取这个文件 |

## 六、命名对照速查表

| 分类 | 命名方式 | 示例 |
|------|----------|------|
| 项目根目录 | kebab-case（小写+连字符） | `campus-activity-system` |
| Java 包名 | 小写，域名倒序+项目名 | `com.cas` |
| Java 类名 | PascalCase（大驼峰） | `ActivityController`, `UserServiceImpl` |
| Java 接口 | PascalCase，不加 I 前缀 | `UserService`（不是 `IUserService`） |
| Java 实现类 | 接口名 + Impl 后缀 | `UserServiceImpl` |
| 数据库表名 | snake_case（小写+下划线） | `sign_ins`, `current_participants` |
| 数据库列名 | snake_case | `real_name`, `start_time` |
| Vue 组件名 | PascalCase | `ActivityDetail.vue` |
| Vue 路由路径 | kebab-case | `/my-activities`, `/activity/:id` |
| JS 文件名 | kebab-case 或 camelCase | `request.js`, `vite.config.js` |
| JS 函数名 | camelCase（小驼峰） | `getActivityDetail`, `handleLogin` |
| SQL 文件 | snake_case | `schema.sql`, `data.sql` |
| 文档文件 | 中文 | `网络实训报告.md` |
| 目录名 | 小写英文 | `controller`, `frontend`, `docs` |

## 七、不在这个项目里的命名惯例（对比参考）

| 你没用但很常见 | 为什么你没用 |
|---------------|-------------|
| `IUserService` | Java 社区以前流行接口加 `I` 前缀，现在主流做法是不加——Spring 官方代码就不加 |
| `UserControllerImpl` | Controller 不需要接口+实现分离，只有一个实现就够了 |
| `user.service.js` | 前端习惯用 `.service.ts` 后缀（Angular 风格），Vue 社区更喜欢按功能分目录 |
| `UserModel.java` | 有些项目把实体叫 Model，但 MyBatis-Plus 文档统称 Entity |
| `DAO` / `Repository` | DAO（Data Access Object）是 JDBC 时代术语；Repository 是 Spring Data JPA 术语。MyBatis-Plus 叫 Mapper |

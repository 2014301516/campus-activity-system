# 《网络软件开发综合实训》课程设计报告

---

## 一、项目简介

本次课程设计项目名称为“校园活动管理系统”。系统面向校园活动组织与参与场景，围绕“活动发布、活动审核、活动报名、活动签到、活动评价”构建完整业务流程，覆盖学生、组织者、管理员三类角色，适用于校园文化活动、讲座报名、志愿服务、社团活动等业务场景。

项目采用前后端分离架构实现，前端基于 Vue 3、Vite、Element Plus、Vue Router、Pinia、Axios 和 ECharts 构建，后端基于 Spring Boot、MyBatis-Plus、Spring Security、JWT 和 MySQL 实现。系统支持角色鉴权、活动状态自动流转、数据统计展示、活动分类管理、公告发布等功能，能够较完整地体现网络软件开发综合实训中对页面、接口、数据库和业务逻辑的综合要求。

开发本项目的背景在于：学校中常见的活动报名与管理流程通常分散在不同平台中，存在信息不统一、审批不透明、签到数据不清晰等问题。通过开发该系统，可以将活动信息展示、流程管理与后台治理集成到同一平台中，既满足课程设计对完整业务系统的要求，也具备一定的实际应用价值。

## 二、项目任务

本次实训的主要任务是独立完成一个具备真实业务流程的 Web 系统，要求系统包含多个页面、至少 4 张核心数据表、清晰的角色权限区分，以及完整的数据增删改查功能。结合课程要求，本项目的开发目标如下：

1. 设计并实现一个完整的校园活动管理系统。
2. 系统需支持多角色登录，并根据角色展示不同页面与功能入口。
3. 学生角色需能够完成活动浏览、报名、取消报名、签到、签退和评价。
4. 组织者角色需能够创建、编辑、删除并提交审核活动。
5. 管理员角色需能够完成活动审核、用户管理、分类管理、公告管理和数据统计。
6. 数据库需围绕用户、活动、报名、签到、评价、公告等核心对象进行设计。
7. 前后端需通过接口联调完成业务闭环，并保证基本的权限控制与错误处理。
8. 最终形成一套包含代码、数据库脚本、项目说明和课程报告的完整成果。

从开发范围上看，本项目重点实现校园活动平台的核心流程，不包含支付、消息推送、短信验证等扩展能力；从整体要求上看，系统强调功能完整、页面清晰、逻辑准确、适合现场演示与答辩。

## 三、项目实现

### 3.1 需求分析

#### 1. 用户角色分析

系统包含三类主要用户：

- 学生：系统的主要使用者，负责浏览活动、报名参加活动、签到签退、查看个人报名记录和提交活动评价。
- 组织者：负责发布和维护活动，设置活动名称、时间、地点、人数上限、封面图和分类信息，并提交管理员审核。
- 管理员：负责平台治理，包括活动审核、用户管理、分类管理、公告管理以及后台统计分析。

#### 2. 业务场景分析

围绕校园活动的实际使用场景，系统需要解决以下问题：

- 活动信息分散，学生无法快速查看近期可参加的活动。
- 活动创建后需要有审核流程，避免不合规内容直接发布。
- 报名成功后缺少统一的签到与签退记录，难以统计实际参与情况。
- 管理员需要从平台整体角度掌握活动数量、状态分布和用户使用情况。

#### 3. 功能需求分析

根据上述场景，系统需要实现以下核心功能：

- 用户注册、登录、身份识别与权限控制。
- 活动列表展示、活动详情查看、分类筛选和关键字搜索。
- 学生活动报名、取消报名、签到、签退和评价。
- 组织者活动创建、编辑、删除、提交审核和查看报名情况。
- 管理员审核活动、管理用户、维护分类、维护公告、查看统计图表。
- 活动状态自动流转，如活动开始后自动变为进行中，活动结束后自动变为已结束。

#### 4. 使用场景分析

- 学生登录后，在首页浏览推荐活动，并进入详情页报名参加活动。
- 学生在活动开始前后一段时间内签到，活动结束前完成签退。
- 组织者创建一场活动，提交审核，等待管理员通过后对外开放报名。
- 管理员进入后台查看待审核活动，并根据活动内容执行通过或驳回操作。
- 管理员查看活动状态分布和平台统计信息，用于汇报或管理分析。

综上，系统不仅需要具备基本的页面展示与数据管理能力，还需要具备较完整的角色权限设计、业务状态控制和前后端联动逻辑。

### 3.2 功能设计和模块划分

结合需求分析，本系统划分为以下几个主要模块：

#### 1. 用户与认证模块

负责完成用户注册、登录、身份识别、Token 鉴权、个人信息查询与修改。该模块是所有业务功能的基础。

#### 2. 首页与活动展示模块

负责展示推荐活动、最新公告、即将开始活动、新上架活动以及活动广场，帮助用户快速发现活动内容。

#### 3. 活动管理模块

负责活动的创建、编辑、删除、详情查看、分类归属、状态维护等，主要面向组织者和管理员。

#### 4. 报名与签到模块

负责学生的活动报名、取消报名、签到、签退和报名记录查询，是系统最核心的业务模块之一。

#### 5. 评价模块

负责学生在参加活动后提交评分和评价内容，用于补充活动反馈信息。

#### 6. 后台治理模块

负责管理员的数据概览、活动审核、用户管理、分类管理和公告管理，是平台管理能力的体现。

通过上述模块划分，系统业务结构更加清晰，便于进行页面设计、接口实现和数据库建模。

### 3.4 难点分析

在项目开发过程中，主要遇到了以下几个技术难点：

#### 1. 多角色权限控制

问题现象：不同角色可以访问的页面和接口不同，如果只在前端做限制，容易被绕过；如果只在后端做限制，前端交互体验会较差。

解决方案：前端通过 Vue Router 根据用户角色控制路由访问和菜单显示；后端通过 Spring Security 与 JWT 对接口进行身份认证和权限控制，实现双重校验。

#### 2. 活动状态自动流转

问题现象：活动创建后会经历待审核、已通过、进行中、已结束等状态，若全部手动切换，不利于演示且容易出现逻辑不一致。

解决方案：在后端统计或查询环节补充状态刷新逻辑，根据当前时间自动将 `approved` 更新为 `ongoing`，将 `approved / ongoing` 更新为 `ended`，减少人工干预。

#### 3. 报名、签到与签退数据一致性

问题现象：若只记录报名人数而不记录真实报名和签到数据，会导致后台列表和前台状态显示不一致，例如人数不为空但报名名单为空。

解决方案：围绕 `activities`、`registrations`、`sign_ins` 三张表建立完整关联关系，前端通过接口查询真实签到状态，不再依赖页面本地猜测状态。

#### 4. 前端交互细节优化

问题现象：在“我的报名”页面中，签到状态与操作按钮容易混淆，出现“看起来能点但其实只是状态提示”的假按钮，影响演示体验。

解决方案：将“签到状态”与“操作”分列处理。状态列只显示 `未签到 / 已签到 / 已签退`，操作列只显示 `签到 / 签退 / 取消报名` 等真实可执行动作，从而使界面语义更加清晰。

#### 5. 中文数据和 SQL 导入问题

问题现象：在部分终端环境下直接写入中文 SQL 数据时出现乱码或问号，影响演示数据质量。

解决方案：统一采用 UTF-8 编码维护 SQL 文件，优先通过脚本文件导入数据库，并在数据库连接参数中开启 `useUnicode=true` 和 `characterEncoding=utf-8`。

通过对这些问题的排查与修复，我进一步掌握了从需求分析、数据库设计、接口联调到页面优化的完整开发过程。

## 四、实训总结

通过本次《网络软件开发综合实训》课程设计，我完成了一个较完整的前后端分离项目，对 Web 系统开发流程有了更加系统的认识。与只完成静态页面或单一模块不同，本项目要求从需求分析、数据库设计、接口开发、权限控制到页面交互全部打通，因此对综合开发能力有较强锻炼作用。

在技术层面，我进一步熟悉了 Vue 3 的组件化开发方式、Vue Router 的路由控制、Pinia 的状态管理、Axios 的接口封装，以及 Spring Boot、MyBatis-Plus 和 Spring Security 在后端业务开发中的应用。同时，我也对 JWT 鉴权、RESTful 接口设计、关系型数据库建模、状态流转设计等有了更深入的理解。

在项目实现过程中，我体会到以下几点：

1. 需求分析决定系统上限。只有先把角色、流程和状态梳理清楚，后续数据库和接口设计才不容易返工。
2. 数据一致性非常重要。页面展示的数据必须与数据库中的真实记录对应，否则系统看起来“能用”，但实际逻辑并不可靠。
3. 权限控制必须前后端协同。单纯依赖某一端都不够稳妥。
4. 演示型项目也要重视细节，例如按钮文案、状态提示、数据完整度，这些都会直接影响老师对系统完成度的判断。

当然，项目目前也还存在一些不足：

- 缺少更完善的日志审计与异常监控机制。
- 页面视觉设计还有继续优化的空间。
- 没有加入文件上传、消息提醒、邮件通知等扩展功能。
- 自动化测试覆盖较少，主要依赖人工联调验证。

后续学习规划方面，我希望继续加强以下方向：

- 深入学习 Spring Security、JWT 和 RBAC 权限模型设计；
- 学习更规范的接口文档与自动化测试工具；
- 尝试部署项目到服务器，掌握前后端上线流程；
- 继续提升数据库优化、代码重构和系统架构设计能力。

总体而言，本次实训不仅帮助我掌握了完整 Web 项目的开发流程，也让我在分析问题、修复问题和整理文档方面得到了较大提升。

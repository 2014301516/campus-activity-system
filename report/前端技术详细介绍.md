## 一、Vue 3

### 是什么
前端框架。把页面拆成一个个 `.vue` 组件，每个组件包含自己的 HTML（`<template>`）+ JS（`<script setup>`）+ CSS（`<style scoped>`）。

### 为什么用它

传统的 HTML+JS+jQuery 写页面，DOM 操作散落各处，页面一复杂代码就乱。Vue 的核心是**数据驱动**——你只管改数据，页面自动更新。比如 `activityList.value = 新数据` 一行代码，列表就刷新了，不需要手动 `$('#list').html(...)`。

### 项目里的用法
7 个页面文件都是 `.vue` 文件，使用 Composition API：
- `ref()` — 创建响应式数据（数据变了页面自动更新）
- `reactive()` — 创建响应式对象
- `computed()` — 计算属性（依赖其他数据，自动缓存）
- `onMounted()` — 生命周期钩子（页面加载时执行）

### 答辩要点
- 响应式数据：数据变 → 视图自动变，不用手动操作 DOM
- 组件化：每个页面是独立组件，复用和维护都方便
- Composition API：比旧版 Options API 逻辑聚合更好，相关代码写在一起

## 二、Vite

### 是什么
**前端构建工具。把 `.vue` 文件编译**成浏览器能运行的 JS/CSS，**开发时启动一个本地服务器**。

### 为什么用它

以前的 Webpack 项目一大就慢，改一行等几秒。Vite 利用浏览器原生 ES Module，**秒级热更新**，改了代码页面几乎立刻刷新。打包用 esbuild/rolldown，速度极快。

### 项目里的配置 `frontend/vite.config.js`

```javascript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { fileURLToPath, URL } from 'node:url'

export default defineConfig({
  plugins: [vue()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url))  // @ → src 目录
    }
  },
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://localhost:8080',  // 代理后端
        changeOrigin: true
      }
    }
  }
})
```

### 常用命令
```bash
npm run dev    # 启动开发服务器 → http://localhost:5173
npm run build  # 生产打包 → dist/
```

### `vite.config.js` 逐行讲解

这个文件只有 22 行，定义了 Vite 的所有行为。

```javascript
// 第 1-3 行：引入依赖
import { defineConfig } from 'vite'            // Vite 的配置函数
import vue from '@vitejs/plugin-vue'            // Vue 3 编译插件
import { fileURLToPath, URL } from 'node:url'   // Node.js 路径工具

export default defineConfig({    // defineConfig 提供 IDE 代码补全

  // 第 7 行：插件
  plugins: [vue()],              // 告诉 Vite："这是 Vue 项目，帮我编译 .vue 文件"
                                  // 没有它，Vite 不认识 <template> 和 <script setup>

  // 第 8-11 行：路径别名
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url))
                                  // 把 @ 映射到 src/ 目录
                                  // import X from '@/api' 等价于 import X from './src/api'
    }
  },

  // 第 13-21 行：开发服务器
  server: {
    port: 5173,                   // 开发服务器端口：http://localhost:5173
    proxy: {                      // 代理规则
      '/api': {                   // 当前端请求 /api/xxx 时
        target: 'http://localhost:8080',  // 转发到后端 8080
        changeOrigin: true                // 伪装请求来源
      }
    }
  }
})
```

### 三个配置分别解决什么问题

| 配置 | 解决的问题 | 怎么说 |
|------|-----------|--------|
| `plugins: [vue()]` | Vite 不认识 `.vue` 文件 | 告诉 Vite：这是 Vue 项目 |
| `alias: { '@': './src' }` | `../../../components/X.vue` 长又容易错 | `@/components/X.vue` 始终从 src 根目录开始，移文件不用改 import |
| `proxy: { '/api': '...' }` | 前端 5173 请求后端 8080 → 跨域报错 | Vite 在中间转发，浏览器以为自己请求的就是 5173 |

### 代理的工作原理（最关键的部分）

```
前端代码写：request.get('/api/user/info')
       ↓
浏览器实际发：http://localhost:5173/api/user/info   ← 注意是 5173
       ↓
Vite 开发服务器收到了 → 匹配到 proxy 规则 /api
       ↓
Vite 帮你转发：http://localhost:8080/api/user/info   ← 服务器之间的请求没有跨域限制
       ↓
后端返回数据 → Vite 返回给浏览器
```

浏览器完全不知道代理的存在，跨域问题被 Vite 悄悄解决了。这个代理只在开发时有效，生产环境需要 Nginx 或后端直接提供前端静态文件。

### `@` 别名的作用

```javascript
// 不用别名：到处是 ../../../
import { userApi } from '../../../api'
import Home from '../../views/Home.vue'

// 用别名：始终从 src 根目录出发
import { userApi } from '@/api'
import Home from '@/views/Home.vue'
```

无论当前文件嵌套多深，`@` 始终指向 `src/` 目录。

### 常用命令
```bash
npm run dev    # 启动开发服务器 → http://localhost:5173
npm run build  # 生产打包 → dist/
```

### 答辩要点
- 热更新极快，开发体验好
- 内置代理功能解决了开发时的跨域问题
- 比 Webpack 配置少、速度快
- `@` 别名避免深层相对路径

## 四、Vue Router

### 是什么
**Vue 的前端路由。让单页应用能像多页网站一样切换页面，**URL 会变但页面不刷新。

### 为什么用它

没有路由的话，页面切换只能靠 `v-if` 判断，**URL 不会变**，无法收藏/分享链接，也不能用**浏览器的前进后退**。

### 项目配置 `frontend/src/router/index.js`

```javascript
const routes = [
  { path: '/login',           component: Login.vue,         meta: { noAuth: true } },
  { path: '/home',            component: Home.vue },
  { path: '/activity/:id',    component: ActivityDetail.vue },  // 动态参数
  { path: '/my-activities',   component: MyActivities.vue },
  { path: '/manage',          component: ActivityManage.vue,
                              meta: { roles: ['organizer', 'admin'] } },
  { path: '/admin',           component: Admin.vue,
                              meta: { roles: ['admin'] } },
  { path: '/profile',         component: Profile.vue },
]

// 路由守卫：没登录 → 强制跳登录页
router.beforeEach((to, from, next) => {
  if (to.meta.noAuth) return next()
  if (!localStorage.getItem('token')) return next('/login')
  next()
})
```

### 页面与 URL 对应

| URL | 页面 | 谁可见 |
|-----|------|--------|
| `/#/login` | Login.vue | 所有人 |
| `/#/home` | Home.vue | 所有人 |
| `/#/activity/3` | ActivityDetail.vue | 所有人 |
| `/#/my-activities` | MyActivities.vue | 学生、组织者 |
| `/#/manage` | ActivityManage.vue | 组织者 |
| `/#/admin` | Admin.vue | 管理员 |
| `/#/profile` | Profile.vue | 所有人 |

### 答辩要点
- 前端路由实现页面切换，无需服务端参与
- 路由守卫实现权限控制，未登录用户看不到内部页面
- 动态路由 `/activity/:id` 通过 `useRoute().params.id` 获取参数

## 五、Pinia

### 是什么
Vue 3 官方推荐的状态管理库。用来存"多个页面都需要用到的数据"。

### 为什么用它

没有状态管理的话，用户信息在每个页面都得单独请求一次，或者通过组件层层传递。Pinia 把数据集中起来，任何页面都能直接读。

### 项目配置 `frontend/src/store/auth.js`

```javascript
export const useAuthStore = defineStore('auth', () => {
  const token = ref(localStorage.getItem('token') || '')
  const userInfo = ref(JSON.parse(localStorage.getItem('userInfo') || 'null'))

  const isLoggedIn = computed(() => !!token.value)
  const role = computed(() => userInfo.value?.role || 'student')

  function setAuth(data) {
    token.value = data.token
    userInfo.value = { userId, username, realName, role }
    localStorage.setItem('token', data.token)
    localStorage.setItem('userInfo', JSON.stringify(userInfo.value))
  }

  function logout() {
    token.value = ''
    userInfo.value = null
    localStorage.removeItem('token')
    localStorage.removeItem('userInfo')
  }

  return { token, userInfo, isLoggedIn, role, setAuth, logout }
})
```

### 哪些文件用了它

| 文件 | 用途 |
|------|------|
| App.vue | 读 `role` → 决定菜单显隐；读 `userInfo.realName` → 显示用户名头像 |
| Login.vue | 登录成功后调 `setAuth()` → 存 token + userInfo |
| ActivityDetail.vue | 读 `isLoggedIn` → 判断显示"登录后参与"还是报名按钮 |
| Profile.vue | 读 `userInfo` → 展示个人信息 |

### 答辩要点
- 比 Vuex 简洁（Vue 3 官方推荐替代 Vuex）
- 支持 Composition API 写法
- localStorage 持久化：刷新页面不影响登录状态

## 六、Axios

### 一句话概括

Axios 是前端和后端之间的"快递员"——前端想拿数据、提交数据，都通过它发 HTTP 请求到后端。

封装了HTTP请求，实现Ajax的npm包

### 是什么

HTTP 请求库。前端用它向后端发请求拿数据。所有 AJAX 请求都通过它。

### Axios 和 Ajax 是什么关系

**Ajax 是技术概念，Axios 是具体工具。**

| | Ajax | Axios |
|------|------|------|
| 是什么 | **技术方案**——Asynchronous JavaScript And XML，表示"用 JS 异步发请求，不刷新页面" | **JS 库**——封装了 Ajax 能力的 npm 包 |
| 层级 | 概念层 | 实现层 |
| 类比 | "发快递"这个行为 | "顺丰"这家具体的快递公司 |

**实现 Ajax 有 3 种方式**：

```javascript
// 方式 1：原生 XMLHttpRequest（最原始，写法繁琐）
const xhr = new XMLHttpRequest()
xhr.open('POST', '/api/user/login')
xhr.send(JSON.stringify(data))

// 方式 2：原生 fetch（现代浏览器自带，比 XHR 简洁但还是要手动处理很多）
fetch('/api/user/login', { method: 'POST', body: JSON.stringify(data) })

// 方式 3：Axios 库（我们项目用的，封装了底层细节，最方便）
await userApi.login(data)
```

**三者都是 Ajax，只是实现方式不同。** 我们选 Axios 是因为它封装了拦截器、超时、自动 JSON 转换，写起来最快。

### 为什么用它

浏览器原生的 `fetch()` API 用起来繁琐，Axios 封装了拦截器、自动 JSON 转换、超时控制、错误处理。

### "request.js 是对 Axios 的统一封装"是什么意思

Axios 是通用工具，request.js 是按我们项目需求定制过的 Axios。

**类比**：买来一部手机（Axios），先贴膜、装 SIM 卡、设锁屏密码。之后每次用手机，这些东西都已经在了，不用重新弄。request.js 就是对 Axios 的"开箱配置"。

**具体来说，request.js 做了 4 项定制**：

```javascript
// 不用 request.js：每次发请求都要从头写一堆重复配置
axios.post('http://localhost:8080/api/user/login', data, {
  headers: { 'Authorization': 'Bearer ' + token },
  timeout: 15000
})

// 用 request.js：把重复的东西提前配好
const request = axios.create({
  baseURL: '/api',       // ① 统一前缀——不用每次写完整 URL
  timeout: 15000         // ② 统一超时——15 秒没响应就报错
})
// ③ 请求拦截器——每次请求自动从 localStorage 取出 JWT 放进 Header
// ④ 响应拦截器——后端报错自动弹提示，401 自动跳登录
```

之后所有页面引用这个 `request` 而不是原生 `axios`，四项配置自动生效：

```javascript
// 页面代码——什么额外参数都不用传，配置全在 request.js 里生效了
const res = await request.post('/user/login', data)
// baseURL 自动拼接 → /api/user/login
// Token 自动挂载
// 超时 15s 自动生效
// 错误自动弹提示
```

**如果不封装**：7 个页面各自 `import axios from 'axios'`，每个都要写一遍 baseURL、Token、错误处理。某天后端把 `/api` 改成 `/v2`，要改 7 个文件。封装后只改 request.js 一处。

### 不用 Axios vs 用 Axios

**浏览器原生 fetch 写法**（每个请求都要写一堆重复代码）：

```javascript
fetch('http://localhost:8080/api/user/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ username: 'student01', password: '123456' })
})
.then(res => res.json())
.then(data => {
  if (data.code !== 200) alert(data.message)     // 每个请求都要写一遍错误判断
})
.catch(() => alert('网络错误'))                    // 每个请求都要写一遍异常处理
```

**用 Axios（项目实际写法）**——配置一次，全局生效：

```javascript
// request.js 配一次
const request = axios.create({ baseURL: '/api', timeout: 15000 })
// 请求拦截器、响应拦截器各写一次

// 页面里调用——一行搞定
const res = await userApi.login({ username: 'student01', password: '123456' })
```

### 项目里它帮我们做了 3 件事

| 功能 | 代码位置 | 效果 |
|------|----------|------|
| 自动挂 Token | `request.js` 请求拦截器 | 每次请求自动从 localStorage 取 JWT 放进 Header，页面不用手动传 |
| 统一错误处理 | `request.js` 响应拦截器 | 后端返回错误 → 自动弹提示；code=401 → 自动清登录跳转 |
| 代理跨域 | `vite.config.js` proxy | `/api` → `localhost:8080`，开发时不报跨域错误 |

### 项目配置 `frontend/src/api/request.js`

```javascript
const request = axios.create({
  baseURL: '/api',        // 所有请求自动加前缀 → Vite 代理到 localhost:8080
  timeout: 15000          // 15 秒超时
})

// 请求拦截器：自动挂 Token
request.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) config.headers['Authorization'] = `Bearer ${token}`
  return config
})

// 响应拦截器：统一处理错误
request.interceptors.response.use(
  response => {
    if (response.data.code !== 200) {
      ElMessage.error(response.data.message)
      if (response.data.code === 401) {
        // Token 过期 → 清登录 → 跳登录页
        localStorage.removeItem('token')
        window.location.hash = '#/login'
      }
      return Promise.reject(new Error(response.data.message))
    }
    return response.data
  },
  error => {
    ElMessage.error('网络错误，请稍后重试')
    return Promise.reject(error)
  }
)
```

### 答辩要点
- 请求拦截器：统一挂 Token，不用每个接口手动加
- 响应拦截器：统一处理错误和 401，不用每个页面重复判断
- 配合 Vite 代理，`baseURL: '/api'` → 开发环境自动转发到 localhost:8080
- 比原生 fetch 省 80% 的重复代码：baseURL、Token、错误、超时全集中配置

## 七、ECharts

### 是什么
百度开源的数据可视化图表库。支持折线图、柱状图、饼图、热力图等几十种图表。

### 为什么用它

不用 Canvas 从零画图，配置几个选项就能出专业图表。

### 项目里的用法

`Admin.vue` → 数据概览 Tab → 饼图

```vue
<script setup>
import { PieChart } from 'echarts/charts'
import { CanvasRenderer } from 'echarts/renderers'
import { use } from 'echarts/core'
use([PieChart, CanvasRenderer])
</script>
```

展示活动状态分布（草稿/待审核/已通过/进行中/已结束/已取消各多少个），数据从 `/api/dashboard/stats` 接口实时获取。

### 答辩要点
- 跟 Element Plus 配合使用，图表和表格风格统一
- 饼图直观展示活动状态占比
- 数据从后端实时接口获取，刷新即更新

    └── <router-view>
        ├── Login.vue          (/#/login)

## 九、数据流

```
用户操作页面
  │
  ├─ Vue 组件调用 @/api/index.js 里的 API 函数
  │
  ├─ request.js Axios 实例自动挂 Token 到 Header
  │
  ├─ HTTP 请求 → Vite 代理 (/api → localhost:8080)
  │
  ├─ Spring Boot Controller → Service → Mapper → MySQL
  │
  ├─ 返回 { code: 200, message: "操作成功", data: {...} }
  │
  ├─ response 拦截器：code≠200 弹错误 / 401 跳登录
  │
  ├─ 返回 data 给调用方 → 改 ref 值
  │
  └─ Vue 响应式系统自动更新页面 DOM
```

## 十、前后端协作标准约定

| 约定 | 前端 | 后端 |
|------|------|------|
| 接口前缀 | `baseURL: '/api'` | `@RequestMapping("/api")` |
| 日期格式 | `value-format="YYYY-MM-DD HH:mm:ss"` | `@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")` |
| 认证方式 | Header: `Authorization: Bearer <token>` | `JwtAuthFilter` 解析 Bearer Token |
| 返回格式 | 响应拦截器校验 `code === 200` | `Result<T>`：`{ code, message, data }` |
| 分页格式 | `{ page, size }` 传参 | `Page<T>` 返回 `{ records, total, pages }` |
| 跨域 | Vite proxy 转发 | `CorsConfig` 允许所有来源 |

### 4. 公共层 `common/`

**目录职责**：存放全项目共用的类——统一返回格式和全局异常拦截。被所有 Controller 依赖。

| 文件 | 作用 | 关键注解 |
|------|------|----------|
| `Result.java` | 统一返回格式 `{ code, message, data }`。所有接口都用它包裹返回值。`success(data)`、`error(msg)`、`unauthorized(msg)` 等工厂方法 | — |
| `GlobalExceptionHandler.java` | `@RestControllerAdvice` 全局异常处理。捕获 RuntimeException → 返回友好提示给前端；捕获其他 Exception → 返回"系统内部错误" | `@RestControllerAdvice` `@ExceptionHandler` |

### 4.1 为什么需要公共层

没有 common 层的项目，每个 Controller 各自写返回格式和异常处理，十个接口十种返回格式：`{success: true}`、`{status: "ok"}`、`{code: 0}`——前端对接成本极高。

**Result.java** 统一了格式：
- 成功：`{code: 200, message: "操作成功", data: {...}}`
- 业务错误：`{code: 500, message: "用户名已存在", data: null}`
- 未登录：`{code: 401, message: "请先登录", data: null}`

前端请求拦截器只需判断 `code === 200`，不是就弹错误提示，无需针对每个接口写不同逻辑。

**GlobalExceptionHandler.java** 统一了异常处理：Service 层任何地方 `throw new RuntimeException("xxx")`，不用每个 Controller 方法写 try-catch，全局异常处理器自动捕获 → 返回 `Result.error("xxx")`。

### 5. 配置层 `config/`

**目录职责**：Spring Boot 项目的"开关面板"——安全规则、JWT 过滤器、跨域、分页插件、JSON 格式，全部在这里集中配置。

| 文件 | 作用 | 详细说明 |
|------|------|----------|
| `SecurityConfig.java` | Spring Security 核心配置 | 禁用 CSRF、开启 CORS、无状态会话。放行公开接口（注册登录、GET 活动/分类/评价/公告）。`/api/admin/**` 需要 admin 角色。POST/PUT/DELETE 活动需要 organizer/admin。其他接口需登录。 |
| `JwtAuthFilter.java` | JWT 认证过滤器 | 继承 `OncePerRequestFilter`。每次请求：从 Header 取 `Authorization: Bearer xxx` → 解析 Token → 校验 → 把 userId+role 注入 SecurityContext → 后续 Controller 通过 SecurityUtil 拿到当前用户 |
| `CorsConfig.java` | 跨域过滤器 | 允许所有来源、所有 Header、所有方法。解决前后端分离不同端口的跨域问题 |
| `MyBatisPlusConfig.java` | MyBatis-Plus 分页插件 | 注册 MySQL 分页拦截器，让 `Page<T>` 自动生成正确的 count + limit SQL |
| `JacksonConfig.java` | JSON 序列化配置 | 让 `LocalDateTime` 在 JSON 里格式化为 `yyyy-MM-dd HH:mm:ss`，解决反序列化失败问题 |

### 5.1 为什么需要配置层

Spring Boot 的核心理念是"约定优于配置"，但安全、跨域、日期格式这些跟业务无关的横切关注点，必须集中管理。如果散落在各个 Controller 里，改一个安全规则要翻遍所有文件。

**config 层把横切关注点集中到 5 个文件**：
- **SecurityConfig**：谁可以访问哪些接口，一个文件全搞定
- **JwtAuthFilter**：所有请求进来先过这个过滤器，自动解析 Token
- **CorsConfig**：跨域规则一处配置
- **MyBatisPlusConfig**：分页插件注册一次，全项目生效
- **JacksonConfig**：日期格式全局统一，不用每个 DTO 重复注解

### 6.1 控制层的作用（为什么需要 Controller）

**一句话**：Controller 是大堂前台——接收请求，调 Service 干活，把结果返回给前端。

**它在三层架构中的位置**：

```
浏览器发请求  →  Controller  →  Service  →  Mapper  →  MySQL
  （前端）        （大堂）       （后厨）     （仓库）    （食材）

Controller 自己不写业务逻辑，只负责：接客 → 传菜 → 上菜
```

**Controller 做的 4 件事**（以 `ActivityController.create()` 为例）：

```java
@RestController                      // 标记这是一个控制器
@RequestMapping("/api")              // 所有接口的路径前缀
public class ActivityController {

    @Autowired
    private ActivityService activityService;     // 注入 Service，不 new

    @PostMapping("/activity")                    // ① 接客：POST /api/activity
    public Result<Activity> create(
            @Valid @RequestBody ActivitySaveDTO dto) {  // ② 验票

        Long userId = securityUtil.getCurrentUserId();   // 拿当前登录用户

        // ③ 传菜：调 Service 处理业务，Controller 不写业务代码
        Activity activity = activityService.createActivity(dto, userId);

        return Result.success("创建成功", activity);  // ④ 上菜：统一格式返回
    }
}
```

| 步骤 | 对应代码 | 含义 |
|------|----------|------|
| ① 接客 | `@PostMapping("/activity")` | 声明 HTTP 方法和 URL |
| ② 验票 | `@Valid @RequestBody` | JSON 自动转 Java 对象 + 校验字段 |
| ③ 传菜 | `activityService.createActivity(...)` | 交给 Service 层处理业务 |
| ④ 上菜 | `Result.success(...)` | 统一格式 `{code:200, data:...}` |

**Controller 帮我们省掉了**：

- URL 路由匹配 — 不用手写 `if (path.equals("/api/user/login"))`
- JSON ↔ Java 转换 — `@RequestBody` 自动，不用手动 `JSON.parse()`
- 参数校验 — `@Valid` 自动，不用 `if (dto.getName() == null)`
- 响应封装 — `Result` 统一格式，不用每个方法手写 JSON

**如果不用 Controller（原始 Servlet 写法）**，一个注册接口要写 100 行手动的 JSON 解析 + 校验 + 调 Service + 拼 JSON 响应。**Spring Boot 的 Controller 把 100 行压缩到 10 行。**

**9 个 Controller 的分工**：

| Controller | 管的业务 |
|------------|----------|
| UserController | 注册、登录、个人信息 |
| ActivityController | 活动增删改查 |
| CategoryController | 分类列表 |
| RegistrationController | 报名、取消 |
| SignInController | 签到、签退 |
| ReviewController | 评价 |
| NoticeController | 公告 |
| AdminController | 管理员专用：审核 + 用户管理 + 分类管理 + 公告管理 |
| DashboardController | 数据统计 |

### 7.1 为什么需要服务层

有人会问：Controller 直接调 Mapper 不就行了吗，为什么要中间加一层 Service？

**如果 Controller 直接调 Mapper**：
```java
// Controller 里写业务逻辑——反模式
if (activity.getCurrentParticipants() >= activity.getMaxParticipants()) {
    return Result.error("报名已满");
}
userMapper.insert(registration);  // 直接操作数据库
activityMapper.updateById(activity);
```

问题：
1. **代码重复**：报名逻辑在 ActivityController 和 AdminController 都需要，两个地方各写一遍
2. **事务难管**：报名涉及 insert 报名记录 + update 活动人数，两步必须一起成功，Controller 管不了事务
3. **测试困难**：想测报名逻辑，得启动整个 Spring Boot 发 HTTP 请求

**有了 Service 层**：
```java
// Controller 只传话
registrationService.register(userId, activityId);

// Service 里集中做正事
@Transactional  // 事务：两步要么一起成功，要么一起回滚
public Registration register(Long userId, Long activityId) {
    // 所有校验和业务逻辑在这里
    activityService.lambdaUpdate()
        .setSql("current_participants = current_participants + 1")  // 原子操作
        .eq(Activity::getId, activityId)
        .lt(Activity::getCurrentParticipants, activity.getMaxParticipants())  // 防超卖
        .update();
    this.save(registration);
}
```

Service 层把业务逻辑从 Controller 抽出来，Controller 瘦成一行调用，Service 可以被多个 Controller 复用，事务注解 `@Transactional` 保证数据一致性。

接口 + 实现类分离还有一个好处：将来想换实现方式（比如报名改为 Redis 排队），只需新写一个 impl，Controller 一行不用改。

### 8. 数据访问层 `mapper/` — 7 个

**目录职责**：对接 MySQL 数据库。每个 Mapper 对应一张表，继承 MyBatis-Plus 的 BaseMapper 后自动获得增删改查能力，无需写 SQL。

| 文件 | 对应表 | 继承 |
|------|--------|------|
| `UserMapper.java` | users | `BaseMapper<User>` |
| `ActivityMapper.java` | activities | `BaseMapper<Activity>` |
| `CategoryMapper.java` | categories | `BaseMapper<Category>` |
| `RegistrationMapper.java` | registrations | `BaseMapper<Registration>` |
| `SignInMapper.java` | sign_ins | `BaseMapper<SignIn>` |
| `ReviewMapper.java` | reviews | `BaseMapper<Review>` |
| `NoticeMapper.java` | notices | `BaseMapper<Notice>` |

全部继承 `BaseMapper<T>` → 自动拥有增删改查方法，**不需要写任何 SQL XML 文件**。复杂查询通过 Service 层的 `LambdaQueryWrapper` 构建。

### 8.1 为什么需要 Mapper 层（为什么用 MyBatis-Plus 而不是手写 SQL）

传统 JDBC 方式操作数据库：自己建 Connection → 写 SQL 字符串 → 设参数 → 执行 → 手动遍历 ResultSet → 塞进对象。一个简单查询 20 行代码。

**MyBatis-Plus 的 BaseMapper 把 20 行压缩到 1 行**：
```java
User user = userMapper.selectById(1L);                    // 查单个
List<User> users = userMapper.selectList(wrapper);        // 条件查询
Page<User> page = userMapper.selectPage(pageParam, wrapper); // 分页
userMapper.insert(user);                                  // 新增
userMapper.updateById(user);                              // 更新
userMapper.deleteById(1L);                                // 删除
```

**LambdaQueryWrapper 解决类型安全问题**：SQL 字段名写成字符串，写错了编译不报错，上线才炸。Lambda 写法 `User::getUsername` 在编译期就能发现字段名变更。
```java
// 字符串方式（容易写错，编译不报错）：
wrapper.eq("user_name", keyword)
// Lambda 方式（编译就检查，安全）：
wrapper.eq(User::getUsername, keyword)
```

### 9.1 为什么需要实体类（entity 和 dto 为什么分开）

新手常把 entity 直接当前后端传输对象——Controller 里把 User 对象原样返回给前端。这有两个隐患：

**隐患一：密码泄露**。User 的 `password` 字段存在 entity 里，如果不小心序列化出去，前端就能看到密码哈希。
```java
// 危险：entity 直接返回，password 字段一起出去了
return Result.success(userMapper.selectById(userId));
```

**隐患二：表结构变更影响前端**。数据库加了一个字段 `internal_notes`（内部备注），不想让前端看到，但如果 entity 就是传输对象，这个字段自动就暴露了。

**解决方案：entity 和 dto 各司其职**：
- entity — 只管跟数据库表一一映射
- dto — 只管前端传什么 / 返回什么，字段按需定义

项目里 Activity 实体有 `@TableField(exist = false)` 标注的非数据库字段（organizerName、categoryName），这些是查出来后 Service 手动填充的，用来给前端展示关联数据，但数据库里不存。

### 10. 数据传输对象 `dto/` — 4 个

**目录职责**：前端和后端之间传数据的专用快递箱。LoginDTO 传登录信息，RegisterDTO 传注册信息，ActivitySaveDTO 传活动表单，ActivityQueryDTO 传查询条件。与 entity 不同——entity 映射数据库，dto 只负责 HTTP 层。

| 文件 | 字段 | 校验注解 |
|------|------|----------|
| `LoginDTO.java` | username, password | `@NotBlank` |
| `RegisterDTO.java` | username, password, realName, studentId, phone, email, role | `@NotBlank`、`@Pattern(regexp = "^1[3-9]\\d{9}$")` 手机号正则 |
| `ActivitySaveDTO.java` | id, title, description, coverImage, categoryId, location, startTime, endTime, maxParticipants | `@NotBlank`、`@NotNull`、`@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")` |
| `ActivityQueryDTO.java` | page, size, keyword, categoryId, status, organizerId, sort, includeAll | — |

### 10.1 为什么需要 DTO

DTO 的核心价值有两个：

**1. 前端校验前置**：Controller 用 `@Valid @RequestBody RegisterDTO` 接收参数，校验注解（`@NotBlank`、`@Pattern`）在进入 Service 之前就拦截非法数据。手机号格式不对，Controller 直接返回 400，Service 根本不用管校验。

**2. 避免"万能参数"**：如果不用 DTO，Controller 要么接收一个巨大的 entity（字段太多），要么接收一堆零散的 `@RequestParam`。有了 DTO，登录就是 LoginDTO（两个字段），活动查询就是 ActivityQueryDTO（8 个查询条件），每个接口的参数精确控制在专用 DTO 里。

### 11. 工具类 `util/` — 2 个

**目录职责**：跨模块使用的工具函数。JwtUtil 管 Token 的生成和解析，SecurityUtil 管"当前登录用户是谁"——被 Controller 和 Service 大量调用。

| 文件 | 作用 | 关键方法 |
|------|------|----------|
| `JwtUtil.java` | JWT 工具 | `generateToken(userId, username, role)` 生成 Token；`getClaimsFromToken(token)` 解析；`validateToken(token)` 校验；`isTokenExpired(token)` 是否过期；`getUserIdFromToken(token)` 获取用户 ID |
| `SecurityUtil.java` | 安全工具 | `getCurrentUserId()` 从 SecurityContext 取当前用户 ID；`getCurrentUserRole()` 取当前角色；`requireStudent()` 校验必须是学生 |

### 11.1 为什么需要工具类

JwtUtil 和 SecurityUtil 被 Controller、Service、JwtAuthFilter 多处调用——如果每个地方各自写一遍 Token 解析逻辑，改密钥算法时要改 5 个文件。抽成工具类，改一处全局生效。

**SecurityUtil 还解决了一个框架问题**：Spring Security 的 SecurityContext 是 ThreadLocal 的，直接 `SecurityContextHolder.getContext().getAuthentication()` 太啰嗦。SecurityUtil 包装成 `getCurrentUserId()` 和 `getCurrentUserRole()`，调用方只需一行。

## 前端 `frontend/`

### 1. 配置文件

**目录职责**：前端项目的"身份证和工具箱"——package.json 声明依赖，vite.config.js 配置代理和别名，index.html 是浏览器加载的第一页。

| 文件 | 作用 |
|------|------|
| `package.json` | npm 依赖清单：Vue 3、Element Plus、Axios、Vue Router、Pinia、ECharts 等 |
| `vite.config.js` | Vite 配置：`@` 别名、端口 5173、`/api` 代理到 localhost:8080 |
| `index.html` | 单页应用入口 HTML，Vite 自动注入 JS。  前端入口 HTML， Vue 应用最终挂载到这里 |

### 2. 入口

**目录职责**：前端启动的"点火开关"——main.js 组装所有插件并挂载 Vue，App.vue 画出整个应用的框架壳。

| 文件 | 作用 | 详细 |
|------|------|------|
| `main.js` | 创建 Vue 应用 | 挂载 Pinia + Router + Element Plus + 注册 Element Plus 全部图标 → 挂到 `#app` |
| `App.vue` | 根组件 | 顶部导航栏（`el-menu` 按角色显隐）+ 用户下拉菜单（`el-dropdown`）+ `<router-view>` 切换页面内容。登录页特殊处理：不显示导航栏，直接全屏渲染 |

### 3. 路由 `router/`

**目录职责**：单页应用不需要后端切换页面——前端路由接管 URL，根据路径加载对应的 Vue 组件。守卫拦截未登录用户。

| 文件 | 作用 |
|------|------|
| `router/index.js` | 7 条路由定义 + `beforeEach` 守卫 |

**路由表**：
```
/login            → Login.vue           meta: { noAuth: true }  （不登录也能看）
/home             → Home.vue
/activity/:id     → ActivityDetail.vue   （动态参数 :id）
/my-activities    → MyActivities.vue
/manage           → ActivityManage.vue    （仅 organizer）
/admin            → Admin.vue            （仅 admin）
/profile          → Profile.vue
```

### 3.1 为什么需要前端路由

传统多页网站：每次点击都向服务器发请求 → 服务器返回整个 HTML 页面 → 浏览器白屏刷新。SPA（单页应用）的做法是：页面只加载一次，之后切换页面靠 JS 替换内容，URL 靠前端路由模拟。

Vue Router 做三件事：
1. **URL 映射**：`/home` 显示 Home.vue，`/admin` 显示 Admin.vue，URL 可以收藏
2. **动态参数**：`/activity/3` 解析出 `:id = 3`，传给 ActivityDetail.vue 查询具体活动
3. **路由守卫**：`beforeEach` 检查是否有 token，没登录就跳到 /login——后端 SecurityConfig 做接口级的权限控制，前端路由守卫做页面级的权限控制，两层防护

```
/api/activity/**  → 后端 SecurityConfig 做接口级权限
/#/manage         → 前端 beforeEach 做页面级拦截
```

### 4. 接口层 `api/`

**目录职责**：前端与后端通信的唯一通道。request.js 管"怎么发"（Token、超时、错误处理），index.js 管"发什么"（所有接口函数清单）。页面只需 `import { xxxApi } from '@/api'` 即可调用。

| 文件 | 作用 |
|------|------|
| `request.js` | Axios 实例：`baseURL: '/api'` + 请求拦截器（挂 Token）+ 响应拦截器（统一错误/401） |
| `index.js` | 所有 API 函数：`userApi`、`activityApi`、`registrationApi`、`signInApi`、`reviewApi`、`categoryApi`、`noticeApi`、`adminApi`、`dashboardApi` |

### 4.1 为什么要把请求单独抽一层

如果每个页面各自 `import axios from 'axios'`，七个页面七个 axios 实例，每个都得单独配 baseURL、超时、Token 挂载、错误处理。某天后端把 `/api` 改成 `/v2/api`，要改七个文件。

**request.js 把"怎么发请求"集中到一个文件**：
- baseURL：改一处全局生效
- 请求拦截器：每个请求自动带 Token，页面无需手动传
- 响应拦截器：统一处理错误弹窗和 401 跳登录，页面只需 `try { await xxxApi.xxx() } catch {}`（甚至 catch 也可以空，因为拦截器已经弹过提示了）

**index.js 把"有哪些接口"集中到一个文件**：所有 API 函数按模块分组导出，页面 import 一行即可调用，拼接 URL 的脏活都封装在函数内部。

### 5. 状态管理 `store/`

**目录职责**：集中存放多个页面都需要的数据（当前登录用户信息）。就像项目的"公告板"——Login 页往上面写，App.vue 和各个页面从上面读，不需要层层传参。

| 文件 | 作用 |
|------|------|
| `auth.js` | Pinia Store：存 `token` + `userInfo`。`setAuth()` 登录、`logout()` 退出、`isLoggedIn` 计算属性、`role` 计算属性。同步到 localStorage |

### 5.1 为什么需要状态管理（Pinia）

不用 Pinia 的话，用户登录后，App.vue 导航栏想知道"当前用户叫什么、什么角色"，Login.vue 有这数据但传不过去——两个页面是平级的，没有父子关系。只能每次读 localStorage，又慢又分散。

**Pinia 做了一件事**：提供一个全局共享的数据盒子。Login.vue 登录成功 → `authStore.setAuth(data)` 把数据放进去。App.vue 导航栏 → `authStore.role` 实时读到当前角色，决定菜单显隐。ActivityDetail.vue → `authStore.isLoggedIn` 判断要不要显示"登录后参与"按钮。

而且数据会自动同步到 localStorage，刷新页面不用重新登录。

### 6. 页面 `views/` — 7 个

**目录职责**：用户直接看到和操作的页面。每个 `.vue` 文件包含三个部分——`<script setup>`（逻辑）、`<template>`（结构）、`<style scoped>`（样式）。页面调 api 拿数据，用 Element Plus 组件渲染。

| 文件 | 路由 | 核心功能 |
|------|------|----------|
| `Login.vue` | `/login` | 登录表单 + 注册表单切换。表单校验（用户名非空、密码 6 位、手机号正则、确认密码一致）。登录成功调 `authStore.setAuth()` → 跳首页 |
| `Home.vue` | `/home` | 左侧搜索框+排序下拉。分类按钮筛选。顶部公告滚动条。活动卡片 CSS Grid 响应式 4/3/2/1 列。点击卡片跳详情。底部分页 |
| `ActivityDetail.vue` | `/activity/:id` | 活动详情描述列表。按钮区（按状态+用户状态组合显隐）：立即报名、取消报名、签到、签退、评价。星星评分弹窗。评价列表展示 |
| `MyActivities.vue` | `/my-activities` | 我的报名表格。每行动态按钮（按是否签到/签退/活动状态组合显隐）。点击活动名跳详情 |
| `ActivityManage.vue` | `/manage` | 我发布的活动列表。发布/编辑弹窗（表单+分类下拉+日期选择+人数）。报名名单弹窗。签到记录弹窗。删除确认 |
| `Admin.vue` | `/admin` | 6 个 Tab：①数据概览（统计卡片+分类计数+状态分布进度条+饼图）②用户管理（搜索+表格+禁用/启用）③活动审核（待审核列表+通过/驳回按钮）④全部活动（表格+状态筛选）⑤分类管理（添加+删除）⑥公告管理（发布弹窗+删除） |
| `Profile.vue` | `/profile` | 个人信息展示（描述列表+角色标签）。编辑模式（表单+保存/取消）。调 `userApi.getUserInfo()` + `userApi.updateProfile()` |

### 6.1 为什么需要页面层（views）

views 是用户唯一能"看到"的层——后端代码写得再好，用户只看页面。每个 `.vue` 文件是一个完整的独立页面，包含三部分：

- `<script setup>`：页面逻辑——调 api 拿数据、处理用户操作、管理页面状态
- `<template>`：页面结构——用 Element Plus 组件搭出界面
- `<style scoped>`：页面样式——`scoped` 保证样式只作用于当前页面，不会污染其他页面

7 个页面覆盖了完整的业务闭环：登录 → 浏览活动 → 查看详情 → 报名 → 签到 → 评价 → 管理后台 → 数据统计。

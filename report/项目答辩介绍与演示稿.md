## 1. 开场介绍

老师好，我的项目名称是 **校园活动管理系统**。

这个项目是一个前后端分离的 Web 管理系统，主要面向校园活动的发布、审核、报名和参与场景，系统中一共设计了三类角色：

- 学生
- 组织者
- 管理员

整个系统围绕一条完整业务流程展开：

**组织者创建活动 -> 管理员审核活动 -> 学生报名参加 -> 学生签到和签退 -> 活动结束后评价 -> 管理员后台统计和管理**

我做这个项目的主要目的是把校园活动从“发布”到“参与”再到“后台管理”的整个流程串起来，不只是做静态页面，而是做一个有完整业务闭环的系统。

---

## 2. 项目背景和意义

在校园场景中，很多活动信息往往分散在群聊、通知和表格里，存在几个问题：

- 学生查看活动不方便
- 活动报名和签到记录不统一
- 组织者管理活动效率较低
- 管理员缺少统一审核和统计入口

所以我设计了这个系统，把活动发布、审核、报名、签到、签退、评价和后台统计集中在同一个平台中。

这个项目不仅满足了课程设计对页面、接口、数据库和权限控制的要求，也更接近一个真实的业务系统。

## 3. 项目技术栈

这个项目采用的是前后端分离架构。

### 3.1 前端技术

- Vue 3
- Vite
- Element Plus
- Vue Router
- Pinia
- Axios
- ECharts

### 3.2 后端技术

- Spring Boot
- MyBatis-Plus
- Spring Security
- JWT
- MySQL

### 3.3 技术分工说明

- 前端负责页面展示、交互和接口调用
- 后端负责业务逻辑、权限控制和数据库操作
- MySQL 负责存储用户、活动、报名、签到、评价等数据

---

## 4. 项目整体架构

从结构上看，项目主要分为四部分：

1. `frontend`：前端页面和交互
2. `backend`：后端接口和业务逻辑
3. `docs`：数据库脚本和项目说明文档
4. `report`：课程报告和答辩准备文档

整个系统的数据流转可以概括为：

**前端页面发请求 -> Axios 统一封装 -> Vite 代理到后端 -> Controller 接收请求 -> Service 处理业务 -> Mapper 操作数据库 -> Result 统一返回给前端**

这也是我在项目里实现前后端通信的核心方式。

组织者主要负责活动发布和维护，主要功能有：

- 创建活动

- 查看活动报名情况
- 提交活动等待管理员审核

### 5.3 管理员

管理员主要负责平台治理，主要功能有：

- 审核活动
- 管理用户
- 管理分类
- 管理公告
- 查看后台统计信息和图表

## 8. 权限控制和前后端通信

这个项目在权限和前后端通信方面，做了比较完整的设计。

### 8.1 前后端通信

前端使用 Axios 统一封装请求，请求配置集中在 `request.js` 中。

主要包括：

- 设置统一请求地址
- 请求拦截器自动携带 token
- 响应拦截器统一处理错误

开发环境下，前端通过 `vite.config.js` 配置代理，把 `/api` 请求转发到 Spring Boot 后端，所以前后端联调比较方便。

### 8.2 登录认证

登录成功后，后端会生成 JWT token。

前端把 token 保存到本地，之后每次请求都会自动带上：

```text
Authorization: Bearer token
```

后端通过 Spring Security 和 JWT 过滤器识别当前登录用户。

### 8.3 角色权限

系统通过前后端双重方式控制权限：

- 前端通过路由控制不同角色能访问的页面
- 后端通过 Spring Security 控制不同角色能访问的接口

这样可以保证页面和接口的权限边界都比较清晰。

## 13. 开发中遇到的难点

答辩时如果老师问“开发中遇到哪些难点”，我可以这样回答：

### 13.1 多角色权限控制

不同角色可访问的页面和接口不同，所以我在前端用路由控制页面访问，在后端用 Spring Security 控制接口权限，实现双重校验。

### 13.2 活动状态流转

活动不是静态数据，而是会经历待审核、已通过、进行中、已结束等状态，所以后端需要根据时间刷新活动状态，保证逻辑一致。

### 13.3 报名与签到数据一致性

如果只改前端展示数字，后台报名名单和签到记录就会对不上，所以我把报名记录、签到记录和活动人数做了关联处理。

### 13.4 前端交互细节

在“我的报名”页面中，我把状态展示和操作按钮分开，避免出现看起来像按钮但其实只是状态提示的情况，使页面交互更清晰。
